# Async Interview Forms — Spec

**Status:** v0.1.
**Date:** 2026-05-06.
**Supersedes:** the interview-agent design previously drafted under `docs/superpowers/specs/`.

## 1. Context

This is a one-shot tool for collecting input from ~10 colleagues to feed the QA AI automation strategy brief. No agent, no sessions, no resume — just two stateless markdown transformations. The earlier interview-agent design was overbuilt for this scope and has been deleted.

## 2. Workflow

```
[1] generate-form (per persona, run once)
       ↓
forms/qa-engineer.md (template, sent to QA Engineer interviewees)
forms/qa-lead.md
forms/developer.md
forms/release-manager.md
forms/product-owner.md
       ↓
[2] User emails / shares each form to the right participants
       ↓
[3] Each participant fills in frontmatter (their name etc.) + answers in the body
       ↓
[4] Each participant returns the file (email reply, shared drive, whatever)
       ↓
[5] User saves returned files as forms/qa-engineer-<participant_slug>.md, etc.
       ↓
[6] parse-form on each filled file
       ↓
responses/<script_id>-<participant_slug>.json (one per filled form)
       ↓
[7] (future) compiler ingests all responses/*.json → strategy brief
```

The user (sponsor) drives steps 2, 4, and 5 manually — there is no agent. Steps 1 and 6 are the two skills this spec covers.

## 3. Form template shape

Generated by `interview-form-generate`. One template per persona (5 total). The interviewee fills in two regions:

- **Frontmatter** — identity fields. The interviewee replaces `<fill in ...>` placeholders.
- **Body** — under each question, a "Your response:" section the interviewee replaces.

```markdown
---
script_id: qa-engineer
persona: QA Engineer (Individual Contributor)
script_version: 0.2.0
participant_name: <fill in your name>
participant_role: <fill in your role / team>
completed_at: <fill in YYYY-MM-DD when done>
---

# Interview: QA Engineer (Individual Contributor)

## Instructions

- Answer each question below in the **Your response:** section under it.
- Skip questions you don't want to answer or that don't apply — leave the response blank or write `(skip)`.
- Be specific where you can. Concrete examples ("the last test I authored", "the time it took on Tuesday") are more useful than generalities.
- When you're done: fill in the frontmatter at the top, save the file, and send it back.

---

## 1. Warm-up & Context

### Q1. How long have you been on the QA team, and what was your role before?

**Your response:**

> _(write your answer here)_

### Q2. What kind of testing do you primarily own — UI, API, integration, performance, exploratory? `[3b]`

**Your response:**

> _(write your answer here)_

### Q3. On a typical day this past week, roughly what % of your time went to authoring vs. execution vs. verification vs. everything else? `[3d]`

**Your response:**

> _(write your answer here)_

...
```

The form preserves the section headings, question numbering, and tag annotations from the source script. Question IDs are derived as `{script_id}-q{number}` (same scheme as before), but they're not visible in the form — they're recovered by the parser from question position.

## 4. Parser output

`interview-form-parse` reads a filled form and produces a single JSON file. Output shape:

```json
{
  "script_id": "qa-engineer",
  "script_version": "0.2.0",
  "participant_name": "Jane Smith",
  "participant_role": "QA Engineer, Payments team",
  "completed_at": "2026-05-08",
  "answers": [
    {
      "question_id": "qa-engineer-q1",
      "number": 1,
      "tags": [],
      "section_heading": "Warm-up & Context",
      "prompt_text": "How long have you been on the QA team, and what was your role before?",
      "status": "answered",
      "response_text": "About 18 months. Before that I was on the integration test team for 3 years."
    },
    {
      "question_id": "qa-engineer-q2",
      "number": 2,
      "tags": ["3b"],
      "section_heading": "Warm-up & Context",
      "prompt_text": "What kind of testing do you primarily own — UI, API, integration, performance, exploratory?",
      "status": "skipped",
      "response_text": null
    }
  ]
}
```

`status` values in async mode: `answered | skipped`. Empty/blank or `(skip)` → `skipped`. Non-empty → `answered`. There is no `declined`/`not_applicable` distinction; in async mode "skipped" is the union.

The output is keyed by participant filename so the future compiler can ingest many of these and aggregate by `script_id`.

**Zero npm dependencies.** Plain JavaScript ESM (`.mjs`), Node ≥ 18 built-ins only (`node:fs`, `node:path`, `node:test`, `node:assert/strict`).

## 6. Helpers

### `helpers/form/generate_form.mjs`

Function: `generateForm({ scriptPath, outPath })`.

- Reads `interview/<script-file>.md`.
- Parses frontmatter to extract `script_id`, `persona`, `script_version`.
- Walks the body extracting numbered questions, their tags, and section headings.
- Emits a markdown form with:
  - Frontmatter containing `script_id`, `persona`, `script_version` (from the source) plus `participant_name`, `participant_role`, `completed_at` placeholders.
  - An "Instructions" section.
  - One section per source section heading, one `### Q{n}.` per question, with a "Your response:" block.

CLI form: `node helpers/form/generate_form.mjs interview/01-qa-engineer-interview.md forms/qa-engineer.md`.

### `helpers/form/parse_form.mjs`

Function: `parseForm({ formPath })` → returns the parsed object (per §4 shape).

- Reads the filled form.
- Parses frontmatter to extract `script_id`, `script_version`, `participant_name`, `participant_role`, `completed_at`.
- Walks the body, finds each `### Q{n}.` block, extracts the question text + tags + section heading + response text.
- Determines `status` per question: blank or `(skip)` (case-insensitive) → `skipped`; otherwise → `answered`.

CLI form: `node helpers/form/parse_form.mjs forms/qa-engineer-jane-smith.md responses/qa-engineer-jane-smith.json`.

### `helpers/form/shared.mjs`

- `parseFrontmatter(text)` — flat-YAML frontmatter parser (~30 lines).
- `walkQuestions(body)` — yields `{ number, prompt_text, tags, section_heading, response_text }` per question. Used by both helpers.

## 7. Out of scope

- The compiler that ingests `responses/*.json` and produces the strategy brief — separate work.
- Any agent, session state, resume token, conversational refinement, confidence-tagging probe, bias-mitigation skill, git branch automation. (None of these apply to async-only.)
- Privacy/consent flows. Internal-org tool, ~10 colleagues; no formal consent layer.
- A skill for emailing/Slacking the forms. The user does this manually.

## 8. Verification

- Run `node helpers/form/generate_form.mjs interview/01-qa-engineer-interview.md forms/qa-engineer.md`. Open `forms/qa-engineer.md` and confirm: every question from the source script is present, frontmatter has the placeholders, instructions section is intact.
- Hand-fill `forms/qa-engineer.md` (rename copy to `forms/qa-engineer-test.md`), fill in 2 answers and leave 1 blank.
- Run `node helpers/form/parse_form.mjs forms/qa-engineer-test.md responses/qa-engineer-test.json`. Inspect output: confirm 2 `answered` and 1 `skipped`, frontmatter values lifted correctly.

## 9. Tests

- `tests/unit/generate_form.test.mjs` — snapshots the generated form against a known source script in fixtures.
- `tests/unit/parse_form.test.mjs` — parses a known filled form (in fixtures) and asserts on the JSON shape.

Both run via `node --test tests/unit/`.

## 10. What's salvaged from the prior design

- `package.json` (already committed) — ESM mode, zero deps. Reused.
- The decision to use Node + plain JS + JSON storage. Reused.
- The design's `answers.yaml/json` shape — partial reuse: parser output is similar but flatter (no `follow_up_count`, `revised_from`, `response_confidence`, etc., since async can't capture those).
- The interview script frontmatter conventions (Task 2.1 of the deleted plan) — still useful to add, since the form generator reads them. Will be folded into the implementation as a small step.

## 11. Implementation effort estimate

- 2 helpers (`generate_form.mjs`, `parse_form.mjs`, `shared.mjs`) — ~250 lines total.
- 2 unit tests with fixtures — ~150 lines.
- 2 SKILL.md files — ~50 lines each.
- Frontmatter added to the 5 existing interview scripts — mechanical.

Roughly one focused session of work. Substantially smaller than the previous plan.
